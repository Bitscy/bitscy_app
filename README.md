# Bitscy

A Bitcoin-native marketplace for African creative women. List your craft, get paid in Lightning, withdraw straight to your Nigerian bank account.

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](#license) [![Next.js](https://img.shields.io/badge/Next.js-16-black)](https://nextjs.org/) [![TypeScript](https://img.shields.io/badge/TypeScript-strict-blue)](https://www.typescriptlang.org/) [![Tailwind CSS](https://img.shields.io/badge/Tailwind-4-38B2AC)](https://tailwindcss.com/) [![PWA](https://img.shields.io/badge/PWA-installable-5A0FC8)](https://web.dev/progressive-web-apps/)

Bitscy solves a real, lived problem: African creators are structurally excluded from global creative marketplaces. Etsy doesn't cleanly support Nigerian sellers. PayPal blocks Nigerian accounts. Stripe isn't available. International buyers can't pay Nigerian artists directly with their existing tools. Bitcoin Lightning is the only payment system that genuinely doesn't care about nationality, banking jurisdiction, or platform policies.

The protagonist is **Adaeze** — a talented Nigerian artist who paused her craft because she couldn't find buyers or get paid reliably. The product is built for her and the millions of women like her.

## Features

### For sellers (Adaeze)

- **One-tap shop creation** — open a shop with a name and a password. No KYC. No bank account required upfront. Identity is a Nostr key, generated and encrypted client-side.
- **Product listings** — title, description, up to 5 photos, price in naira, category. The price round-trips through sats so buyers anywhere on earth can pay.
- **Lightning-native sales** — buyers pay BOLT-11 invoices generated by the platform Breez wallet. The funds land instantly. The seller's balance updates in real time.
- **Order management** — see incoming orders, view the buyer's shipping address (NIP-04 encrypted, decryptable only by the seller), mark as shipped.
- **NGN withdrawals** — withdraw sats to a Nigerian bank account via Bitnob. Real integration on Bitnob's sandbox: real API, real Lightning testnet routing, no real naira moves in v1.
- **Public storefront** — a shareable URL like `bitscy.com/shop/adaeze-studio` that lists every active piece.

### For buyers (Tobi)

- **Browse without signing up** — `/marketplace` is public; only the checkout step requires an account.
- **Pay with any Lightning wallet** — Phoenix, Wallet of Satoshi, Breez, your hardware wallet. Scan the QR or open in your wallet of choice.
- **Self-custodial embedded wallet** *(stretch)* — for buyers without a Lightning wallet, generate a Bitcoin-backed wallet right in the browser via Breez SDK WASM. The mnemonic never leaves the device.
- **Pseudonymous identity** — Nostr key under the hood; no email, no phone, no real-name requirement.
- **Encrypted shipping** — your shipping address is encrypted to the seller's pubkey via NIP-04. Bitscy never sees the plaintext.

### Architecture pillars

- **Zero-knowledge auth.** PBKDF2 + AES-GCM, 210k iterations. The server never receives the password or plaintext Nostr secret key — only an opaque encrypted blob.
- **Postgres for queries, Nostr for verifiability.** Every product is mirrored as a NIP-15 kind 30018 event. Every order as a kind 30019 event. If Bitscy disappeared tomorrow, the catalog would still exist on public relays.
- **Mobile-first PWA.** Installable to home screen on Android and iOS. Works offline (sans payments). No app stores.
- **Real integrations only.** No mocks in v1: real Breez Lightning, real Bitnob (sandbox), real CoinGecko prices, real Nostr relays, real Cloudinary uploads. The only thing that doesn't actually move is real Naira to a real bank — that's gated behind Bitnob KYB, an explicit v2 step.

## Nostr Implementation

Bitscy implements eight standard NIPs and defines four custom event kinds for marketplace state that the existing NIPs don't cover. The marketplace is interoperable with any NIP-15 / NIP-99-aware Nostr client; the custom kinds are documented so other marketplaces can adopt them.

### Standard NIPs

| NIP | Title | Where it shows up |
|---|---|---|
| [NIP-01](https://github.com/nostr-protocol/nips/blob/master/01.md) | Basic protocol | Event creation, Schnorr signing, relay publishing |
| [NIP-06](https://github.com/nostr-protocol/nips/blob/master/06.md) | Mnemonic key derivation | 12-word BIP39 → Nostr secret key (`m/44'/1237'/0'/0/0`) for recovery |
| [NIP-15](https://github.com/nostr-protocol/nips/blob/master/15.md) | Marketplace | Stalls (kind 30017) and products (kind 30018) |
| [NIP-19](https://github.com/nostr-protocol/nips/blob/master/19.md) | Bech32 encoding | npub / nsec / note encoding throughout auth and display |
| [NIP-23](https://github.com/nostr-protocol/nips/blob/master/23.md) | Long-form content | Seller bios published as kind 30023 articles |
| [NIP-44](https://github.com/nostr-protocol/nips/blob/master/44.md) | Versioned encryption (v2) | Buyer shipping address encrypted to seller's pubkey. Replaces NIP-04, which has known cryptographic weaknesses. |
| [NIP-57](https://github.com/nostr-protocol/nips/blob/master/57.md) | Lightning Zaps | Zap receipts (kind 9735) published on every payment settlement; seller `lud16` in kind 0 profile metadata |
| [NIP-65](https://github.com/nostr-protocol/nips/blob/master/65.md) | Relay list metadata | Each user publishes kind 10002 with their preferred relays |
| [NIP-98](https://github.com/nostr-protocol/nips/blob/master/98.md) | HTTP auth | Login challenge-response via signed kind 27235 events |
| [NIP-99](https://github.com/nostr-protocol/nips/blob/master/99.md) | Classified listings | Products dual-published as kind 30402 for ecosystem interop |

### Custom event kinds (Bitscy-specific)

NIP-15 and NIP-99 define stalls, products, and listings — but leave post-listing lifecycle undefined: order state transitions, reviews, seller reputation, and stall operational status. Bitscy extends the protocol with four parameterized-replaceable event kinds in the unassigned `30050`–`30053` range. Kinds `30054`–`30059` are reserved for future use.

| Kind | Name | Description | Signer |
|---|---|---|---|
| `30050` | `order:state` | Current state of an order (`shipped` / `delivered` / `disputed` / `refunded`). References the order receipt via `e` tag. Latest state wins (parameterized-replaceable on order id). | Seller / Buyer / System |
| `30051` | `product:review` | Buyer feedback on a completed order. 1–5 star rating in tag; markdown body in content. References the order via `e` tag, the product via `a` tag. Updatable once per buyer per order. | Buyer |
| `30052` | `seller:badge` | Bitscy-issued attestation. Carries `firstSaleAt` and `totalSales` — auto-issued from the ledger on first sale, auto-updated on each subsequent sale. No manual ops process. | System |
| `30053` | `stall:status` | Lightweight operational overlay on a stall (`open` / `vacation` / `closed`) without re-publishing the full stall definition. | Seller |

Full spec, schemas, examples, and rationale: [`docs/bitscy-custom-nips.md`](docs/bitscy-custom-nips.md).

## Quick Start

### Prerequisites

- **Node.js** ≥ 22
- **pnpm** 9
- **PostgreSQL** (Supabase, Neon, or local — `DATABASE_URL` + `DIRECT_URL`)
- **Cloudinary** account (free tier; for product image uploads)
- **Breez API key** for Lightning (free, [request here](https://breez.technology/request-api-key/))
- **Bitnob sandbox** account (free, [sandbox.bitnob.co](https://sandbox.bitnob.co)) — required only for the seller withdrawal flow

### Installation

```bash
# 1. Clone
git clone https://github.com/Bitscy/bitscy_app.git
cd bitscy_app

# 2. Install
pnpm install

# 3. Configure
cp .env.example .env.local
# fill in DATABASE_URL, BREEZ_API_KEY, CLOUDINARY_*, SESSION_SECRET, etc.

# 4. Push schema
pnpm db:push

# 5. (Optional) seed demo data
pnpm db:seed

# 6. Run
pnpm dev
```

The app runs at `http://localhost:3000`.

### Generating system keys

```bash
# Generate a 32-byte Nostr secret key (hex) for SYSTEM_NSEC
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# Generate a 32-byte session secret for SESSION_SECRET
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

## Architecture

Four layers, top to bottom.

```
┌──────────────────────────────────────────────────────────────┐
│ 1. PWA client                                                │
│    Installable mobile-first web app on Android & iOS         │
└──────────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────────┐
│ 2. Frontend (Next.js App Router + React 19)                  │
│    Pages, components, Zustand stores, client-side crypto,    │
│    optional embedded Breez wallet (browser WASM).            │
└──────────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────────┐
│ 3. Backend (Next.js API routes)                              │
│    Auth, product CRUD, order state machine, Lightning        │
│    invoice orchestration, Nostr publishing, Bitnob payouts.  │
└──────────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────────┐
│ 4. External infrastructure                                   │
│    Breez SDK Liquid · Bitnob sandbox · CoinGecko · Nostr     │
│    relays · PostgreSQL (Supabase) · Cloudinary · Web Push    │
└──────────────────────────────────────────────────────────────┘
```

### Identity & custody

- **Auth is zero-knowledge.** The client generates a Nostr keypair, derives a 256-bit AES key from the user's password via PBKDF2-SHA256 (210k iterations), encrypts the 12-word recovery mnemonic, and ships `{ npub, encryptedKey, salt, iv }` to the server. No password, no plaintext key.
- **Sellers are platform-custodial in v1.** Funds land in a single Bitscy-operated Breez wallet; per-seller balances are tracked in an append-only `LedgerEntry` table. Sellers can always withdraw to NGN or to a personal Lightning address. v2 migrates each seller to their own self-custodial wallet.
- **Buyers are self-custodial.** Either pay from an existing Lightning wallet (Phoenix, Wallet of Satoshi, etc.) or create a Bitscy-embedded Breez wallet client-side. Mnemonic stays in IndexedDB, encrypted under the user's passphrase.

### Order state machine

```
PENDING ──invoice settled──▶ PAID ──seller marks shipped──▶ SHIPPED ──buyer confirms──▶ DELIVERED
   │                          │                               │
   │                          │                               └──buyer disputes──▶ (disputed Nostr event)
   │                          │                                                         │
   │                          └──invoice expires / buyer abandons──▶ CANCELLED ◀──refund issued──┘
```

Atomic updates (`UPDATE … WHERE status = 'PENDING'`) handle the race between the Breez SDK `paymentSucceeded` event and the frontend polling `/api/lightning/verify`.

Each state transition publishes a kind `30050` Nostr event so the order lifecycle is verifiable on public relays independently of the Bitscy API.

### Authoritative spec

The complete end-to-end architecture lives in [`Bitscy_Integration_Flow.md`](Bitscy_Integration_Flow.md) — the authoritative spec for the whole system.

## Usage

### For sellers

1. Open `/sell` or `/signup` and create your shop with a name + password. Your Nostr identity is generated locally.
2. Save your password — Bitscy cannot reset it.
3. List a product: title, description, up to 5 photos, naira price, category. Photos upload directly to Cloudinary; the listing publishes to Nostr in the same request.
4. Share `bitscy.com/shop/<your-slug>` with buyers.
5. When a sale lands, the dashboard updates instantly: balance, total earned, recent orders.
6. Decrypt the buyer's shipping address (only you can — NIP-44 v2 encrypted to your pubkey), ship the piece, mark as shipped.
7. Withdraw to your Nigerian bank account: pick a saved bank, enter an amount, confirm. Bitscy routes the sats through Bitnob's sandbox; NGN reaches the bank in production.

### For buyers

1. Browse `/marketplace`. Filter by category. No account needed to look.
2. Tap a piece. Read the description. Tap **Buy**.
3. Sign up (if you don't already have an account — it's the same client-side-crypto flow, just with `role: 'BUYER'`).
4. Scan the QR with any Lightning wallet, or tap **Open in wallet** for a lightning: deep-link. Pay.
5. The page detects settlement within seconds and routes you to a success screen.
6. Track the order under `/buyer/orders`.

### Local Lightning testing

The default dev experience uses LNBits (a Lightning backend that supports both real testnet and mock modes). If you don't want to run Lightning at all:

- `USE_MOCK_LIGHTNING=1` skips Breez init entirely; invoices come from a stub
- `LNBITS_ADMIN_KEY` left unset → `src/instrumentation.ts` logs a warning and proceeds without registering the event listener (auth, products, profile still work fine)

For a real settlement test, point Breez at signet or mainnet (start small — `DEMO_BTC_NGN_RATE` keeps the on-screen values matched).

### Database migrations

Two paths:

- **`pnpm db:push`** for the hackathon-velocity flow. Schema changes apply without producing migration files. Fast, but no history.
- **`pnpm db:migrate`** for tracked migrations once the schema stabilizes.

### Tests

```bash
pnpm test           # vitest run
pnpm test:coverage  # with v8 coverage
```

Integration tests under `src/__tests__/commerce/` and `src/__tests__/api/` hit a real Postgres test database (see `src/__tests__/integration-setup.ts`).

## Deployment

### Vercel (recommended)

1. Connect this repo to a Vercel project.
2. Set all environment variables from the [Configuration](#configuration) table.
3. Push to `main` — Vercel runs `prisma generate && next build --webpack` automatically.

The build script intentionally passes `--webpack` because Serwist's default mode is webpack-only ([serwist/serwist#54](https://github.com/serwist/serwist/issues/54)). Don't remove the flag.

### Other platforms

Any Node 22+ host with a Postgres connection works. The app is a stateless Next.js server with a few in-memory caches (CoinGecko rate, Breez SDK instance). Cron jobs (`/api/cron/expire-pending`) need a scheduler — Vercel Cron, GitHub Actions, or your platform's equivalent.
